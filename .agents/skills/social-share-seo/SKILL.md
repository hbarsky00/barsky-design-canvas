# Social Share & SEO Guardrails (barskydesign.pro)

The site is a Vite SPA. Social crawlers (LinkedIn, Facebook, Slack, X, Discord, iMessage) **do not execute JS**, so `react-helmet-async` tags from `UnifiedSEO.tsx` are invisible to them. Previews only work because the Cloudflare worker (`cloudflare/seo-bot-proxy.js`) routes bot user-agents to the `seo-handler` edge function (`supabase/functions/seo-handler/index.ts`), which returns prerendered HTML with the correct `<title>`, meta description, canonical, and `og:*` tags.

If any of those four pieces drift out of sync, sharing breaks silently.

## The four sources of truth (KEEP IN SYNC)

| What | File | Used by |
| --- | --- | --- |
| Static fallback head | `index.html` | Pre-hydration browsers, dumb crawlers that bypass the worker |
| Client-side per-route head | `src/components/seo/UnifiedSEO.tsx` + `src/data/seoData.ts` (`STATIC_PAGE_SEO`, `PROJECT_SEO_MAP`, `BLOG_SEO_MAP`, `BLOG_IMAGE_MAP`) | Browsers, Googlebot |
| Bot prerender | `supabase/functions/seo-handler/index.ts` (`STATIC_SEO`, `PROJECT_SEO`, `BLOG_SEO`) | LinkedIn, Facebook, Slack, X, Discord, iMessage, ChatGPT, etc. |
| Discoverability | `public/sitemap.xml` (generated by `scripts/generate-sitemap.ts`) + `public/robots.txt` | Search engines |

**Rule:** every new page, project, or blog post MUST be added to all three SEO sources (`seoData.ts`, `seo-handler/index.ts`, sitemap entries) in the SAME change. A page that exists in only one place will share with wrong/blank metadata.

## Required fields per entry

- `title` ≤ 60 chars, includes a real keyword (not "Project: foo")
- `description` 50–160 chars, no truncation, no ellipsis
- `image` absolute `https://barskydesign.pro/...` URL, 1200×630, project-specific (never the default profile image as a stand-in)
- canonical always on `https://barskydesign.pro` (never `*.lovable.app`, never preview hosts) — enforced by `SEO_CONSTANTS.BASE_URL`

## Workflow when adding/editing a project or blog post

1. Add/update entry in `src/data/seoData.ts` (`PROJECT_SEO_MAP` or `BLOG_SEO_MAP` + `BLOG_IMAGE_MAP`).
2. Add/update entry in `supabase/functions/seo-handler/index.ts` (`PROJECT_SEO` or `BLOG_SEO`). Edge function auto-deploys.
3. Ensure the OG image exists at the URL referenced. If it's a new image, generate at 1200×630 and place under `public/images/` with a descriptive slug-based filename.
4. Add the route to `scripts/generate-sitemap.ts` `entries` (runs on `predev`/`prebuild`).
5. Verify the four fields match across `seoData.ts` ↔ `seo-handler/index.ts`. They should not diverge.

## Workflow when adding a new static page / route

1. Add route to `src/App.tsx` (or router file).
2. Add entry to `STATIC_PAGE_SEO` in `src/data/seoData.ts`.
3. Add entry to `STATIC_SEO` in `supabase/functions/seo-handler/index.ts`.
4. Add to `scripts/generate-sitemap.ts`.

## What never to change without asking

- `cloudflare/seo-bot-proxy.js` BOT_AGENTS list — removing an entry silently breaks that platform's previews.
- The `barskydesign.pro` canonical base — must stay this exact host.
- `index.html` static `<title>` / `<meta name="description">` — they're the last-resort fallback.
- `Sitemap:` directive in `public/robots.txt` — must point to `https://barskydesign.pro/sitemap.xml`.

## After any SEO change — REQUIRED verification

1. Build passes.
2. `curl -A "facebookexternalhit/1.1" https://barskydesign.pro/<path>` returns HTML containing the expected `<title>` and `og:image`. (Tell the user to run this, or run it yourself if the path is reachable.)
3. Tell the user to **purge Cloudflare cache** (Caching → Purge Everything) after publishing. Without a purge, LinkedIn/Facebook will serve the stale HTML for hours.
4. Tell the user to re-scrape via Facebook OG Debugger / LinkedIn Post Inspector / `https://cards-dev.twitter.com/validator` after the purge.

## Diagnosing "share preview is blank/wrong"

Check in this order:
1. `curl -A "facebookexternalhit/1.1" <url>` — does the bot get the right HTML? If no → seo-handler entry missing/wrong, or Cloudflare worker not deployed.
2. `curl <url>` (no UA) — does the static `index.html` look right? If wrong title there, fix `index.html`.
3. Check `public/sitemap.xml` includes the URL.
4. Check `public/robots.txt` doesn't accidentally `Disallow:` the path.
5. If everything looks right but previews are stale: Cloudflare cache wasn't purged after last deploy. This is the #1 cause.

## Anti-patterns (do not do)

- Adding a project to `seoData.ts` only. Bots will see the generic `index.html` fallback → blank/wrong preview.
- Using `DEFAULT_IMAGE` (Hiram's headshot) as a project's `og:image`. Every project needs its own 1200×630 image.
- Hardcoding `*.lovable.app` URLs anywhere in SEO data.
- Letting `title`/`description` diverge between `seoData.ts` and `seo-handler/index.ts`. They must match.
- Claiming "SEO is fixed" without curling with a bot UA to verify.
